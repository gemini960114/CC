# Claude Code 的檔案讀寫與編輯鏈路

## 為什麼檔案鏈路是 Claude Code 的基本盤

再強的 AI 程式設計工具，如果不能穩定地：

- 讀取檔案
- 理解檔案
- 編輯檔案
- 寫回檔案

那它就只能停留在“建議型助手”的層面。  
Claude Code 之所以真正進入工程工作流，檔案鏈路是最底層的原因之一。

![Claude Code 文件介面](/images/tutorials/claude-code-docs.png)

## 從工具登錄檔看，檔案能力是一級公民

在 `tools.ts` 裡，檔案工具是基礎工具集的一部分：

```
export function getAllBaseTools(): Tools {
  return [
    AgentTool,
    TaskOutputTool,
    BashTool,
    ...(hasEmbeddedSearchTools() ? [] : [GlobTool, GrepTool]),
    ExitPlanModeV2Tool,
    FileReadTool,
    FileEditTool,
    FileWriteTool,
    NotebookEditTool,
    WebFetchTool,
  ]
}
```

這段程式碼說明 Claude Code 不把檔案能力當成附加功能，而是當成預設執行時的核心元件。

## `FileReadTool` 的設計很能說明工程取向

來看它的定義片段：

```
export const FileReadTool = buildTool({
  name: FILE_READ_TOOL_NAME,
  searchHint: 'read files, images, PDFs, notebooks',
  maxResultSizeChars: Infinity,
  strict: true,
  async description() {
    return DESCRIPTION
  },
  isConcurrencySafe() {
    return true
  },
  isReadOnly() {
    return true
  },
  getPath({ file_path }): string {
    return file_path || getCwd()
  },
})
```

這段程式碼至少傳達出 4 個資訊：

1. 它不是隻讀文字，還考慮圖片、PDF、notebook
2. 工具行為是嚴格定義的
3. 它顯式標記自己是隻讀工具
4. 它和當前工作目錄是繫結的

也就是說，Claude Code 對“讀檔案”這件事並沒有寫成一個隨便的 `fs.readFile` 包裝，而是完整地工具化了。

## 為什麼讀檔案工具要這麼複雜

因為真實工程環境裡的“讀檔案”，會立刻碰到很多邊界問題：

- 檔案太大
- 檔案不是純文字
- 路徑需要展開和規範化
- 許可權規則要生效
- 輸出不能把 UI 撐爆

所以一個真正可用的 FileReadTool，必須同時處理內容、路徑、許可權、摘要和上下文。

## 讀檔案和寫檔案不是同一種風險等級

這在 Claude Code 的設計裡很明顯。

- `FileReadTool` 會標記 `isReadOnly()`
- 編輯類工具則會進入更嚴格的許可權和 diff 流程


```mermaid
flowchart LR
    A[開始] --> B[Claude Code 處理]
    B --> C[Tool 執行]
    C --> D[結果輸出]
    D --> E[完成]
```


## 編輯鏈路的關鍵，不只是寫回磁碟

真正重要的是“修改必須可解釋”。  
Claude Code 在檔案編輯這件事上，明顯更強調：

- 變更前後差異可展示
- 許可權審批能看懂修改內容
- UI 能用 diff 呈現

從原始碼引用鏈也能看出來，圍繞 `FileEditTool` / `FileWriteTool` 有專門的許可權請求元件和 diff 元件。

## 為什麼它同時保留 `Edit` 和 `Write`

這通常對應兩種不同的語義：

- `Edit`：對已有內容做定向修改
- `Write`：新建或整體覆蓋檔案

這兩種能力如果混在一起，會帶來兩個問題：

- 許可權模型不清晰
- UI diff 和使用者預期不穩定

把它們拆開，系統才更容易做精細控制。

這個閉環說明：  
檔案工具不是獨立功能，而是主迴圈裡最常見的一類執行節點。

## 真實工程意義

研究 Claude Code 的檔案鏈路，最大的價值不只是看它怎麼改檔案，而是理解一個結論：

> 真正可用的 AI 程式設計系統，必須把“檔案操作”從普通指令碼呼叫，升級成帶語義、帶許可權、帶 UI、帶摘要的系統能力。

這也是為什麼很多“玩具 AI 程式碼助手”很難走遠。

## 小結

Claude Code 的檔案鏈路，本質上是在做三件事：

- 把讀取能力系統化
- 把修改能力視覺化
- 把寫回動作安全化

檔案工具之所以重要，不是因為它們存在，而是因為它們被做成了主迴圈裡穩定可靠的執行基礎設施。
